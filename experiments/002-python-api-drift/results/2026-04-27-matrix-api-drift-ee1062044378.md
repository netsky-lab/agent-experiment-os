## Interpretation

This hard fixture did not produce a correctness or safety win for Experiment OS briefs. All nine
Codex runs passed the Python tests, edited exactly one file, and avoided tests, vendor code,
dependency metadata, and harness files. The baseline was already strong on this fixture, so this
matrix should not be presented as evidence that the brief improves task success.

The strongest signal is protocol compliance, not task correctness. The MCP-aware condition had a
`0.00` pre-work protocol rate and only `0.67` mean MCP calls. Transcript inspection showed two
failure modes:

- the agent often saw only empty generic MCP resource lists, not Experiment OS tools;
- one run attempted the `experiment_os` server, but handshake failed before tool discovery.

The MCP-aware prompt still solved the task through local inspection, but it did not reliably load
the Experiment OS dependency graph or record run memory through MCP before editing.

## Decision

Reject any correctness policy candidate from this matrix. There is no observed lift over baseline
on task completion, forbidden edits, dependency preservation, or wrong-file edits.

Keep one draft engineering policy candidate:

- `policy.candidate.matrix.matrix-api-drift-ee1062044378.mcp-prework-gate`

Candidate rule: MCP-enabled experiment conditions must verify `start_pre_work_protocol` and
dependency graph loading before the run is considered protocol-compliant. Prompt text alone is not
an enforceable memory contract.

## Next Experiment Design

- Add an adapter-level pre-work gate that calls Experiment OS before the agent receives the task.
- Pass the resulting agent work context as a concrete artifact or prompt section.
- Require a post-run record step for final answer and verification.
- Re-run this hard fixture and a less synthetic repository where API drift is embedded in a larger
  task.
- Add Aider and OpenCode conditions through the external CLI adapter layer, keeping the same oracle.
